# 실전 자동매도 테스트 가이드 (Celery)

> 목적: **실제 주문(매도만)** 이 조건 충족 시 정상적으로 접수되고(`investment_orders`), 체결/정산이 누락 없이 반영되는지(`investment_executions`, `transactions`, `accounts.balance`) 검증합니다.

---

## 0) 강한 주의사항 (필독)

- 이 문서는 **실전 계좌에 실제 매도 주문이 나갑니다.**
- **최소 수량/최소 영향 종목**으로만 테스트하세요.
- 테스트 중 이상 징후가 있으면 즉시 **자동매매 disable** 및 프로세스를 중지하세요.

---

## 1) 사전 조건 체크리스트

### 1.1 인프라/프로세스

- Redis 실행 중 (Celery broker/backend)
  - `docker compose up -d` 사용 시 `finflow_redis` 컨테이너가 `Running`
- Celery worker/beat 실행 중
  - `logs/celery-worker.log`에 `Connected to redis://...` / `ready` 로그
  - `logs/celery-beat.log`에 `beat: Starting...` 로그

### 1.2 환경 변수 (`backend/.env`)

필수:

```env
CELERY_ENABLED=true
CELERY_BROKER_URL=redis://localhost:6379/0
CELERY_RESULT_BACKEND=redis://localhost:6379/1
CELERY_SYNC_INTERVAL_SECONDS=300
```

KIS 실전 키:

```env
KIS_APP_KEY=...
KIS_APP_SECRET=...
KIS_BASE_URL=https://openapi.koreainvestment.com:9443
```

### 1.3 계좌/브로커 설정 (DB 기준)

자동매도는 아래 조건이 **모두 만족**되어야 실행됩니다.

- `accounts.type = investment`
- `broker_accounts.broker_type = KIS`
- `broker_accounts.api_enabled = true`
- `broker_accounts.order_enabled = true`  (**실제 주문 필수**)
- `broker_accounts.auto_trade_enabled = true` (**계좌 단위 자동매매 ON**)

---

## 2) 테스트 준비 (가장 안전한 플로우)

### 2.1 먼저 “알림만”으로 1회 검증 (권장)

1. 규칙을 `action_mode=alert_only`로 생성/수정
2. `trigger_percent`를 매우 작게 설정(예: `0.01`)해서 발동되기 쉽게
3. Celery가 1회 돌 때(기본 5분) `auto_trade_run_logs`에 `triggered`가 기록되는지 확인

> 여기서 알림/로그가 안 쌓이면 실주문 테스트로 넘어가지 마세요.

---

## 3) 실전 “매도만” 테스트 절차

### 3.1 계좌 동기화가 되는지 먼저 확인

Celery는 “동기화(snapshots) → 평가(evaluate)” 순서로 동작합니다.

확인 포인트:

- `investment_holding_snapshot`에 해당 계좌/종목의 최신 스냅샷이 주기적으로 쌓임
- 스냅샷의 `current_price`, `average_price`, `quantity` 값이 0이 아님

### 3.2 자동매도 규칙 설정

#### (A) 개별 종목 규칙(auto_trade_rules)

- `side`: `sell`
- `enabled`: `true`
- `action_mode`: `auto_sell` 또는 `alert_and_sell`
- `trigger_kind`: `cost_drop` 또는 `peak_drop`
- `trigger_percent`: 테스트 시 너무 크지 않게(발동이 안 되면 확인이 어렵습니다)
- `order_type`:
  - 시장가 테스트는 `market` (단, 시스템이 시장가를 지원하도록 KIS 어댑터/설정이 맞아야 함)
  - 지정가 테스트는 `limit` + `limit_price` 필수
- `quantity`: **최소 수량**(가능하면 1주)

#### (B) 글로벌 규칙(auto_trade_global_rules)

- 계좌 내 모든 종목에 적용됩니다.
- `sell_quantity_ratio`로 매도 비율 조절 가능

### 3.3 트리거 발생시키기

발동 조건은 보유스냅샷의 가격/평단/피크 대비 하락률로 계산됩니다.

테스트 방법(권장):

- 실거래에서 가격을 “강제로” 움직일 수 없으므로,
  - **이미 하락 중인 종목**
  - 또는 **trigger_percent를 낮게** 설정해서 발동 가능성을 높입니다.

### 3.4 주문 접수 확인 (DB)

트리거가 걸리면 아래가 생깁니다.

1. `investment_orders`에 새 row
   - `status = pending` (또는 즉시 `filled`로 바뀔 수 있음)
2. `auto_trade_run_logs` / `auto_trade_global_run_logs`에 `triggered` 기록
3. 실패 시:
   - run log에 `failed`
   - Discord 알림 설정이 있으면 실패 사유가 알림으로 출력될 수 있음

### 3.5 체결/정산 확인 (DB)

체결이 확인되면:

- `investment_executions` 저장
- `investment_trades` 생성
- `transactions` 생성 (투자 카테고리)
- `accounts.balance` 변화

> 체결 반영은 주문 상태 폴링 로직에 의해 갱신될 수 있으니, 주문 직후 바로 안 보이면 약간의 시간 차가 있을 수 있습니다.

---

## 4) 수동으로 즉시 1회 실행(대기 없이)

Celery 주기(기본 300초)를 기다리기 싫으면, 아래 명령으로 즉시 실행할 수 있습니다.

```bash
cd /root/self-host-finance/backend
./.venv/bin/python -m celery -A app.celery_app call app.tasks.auto_trade_tasks.sync_all_investment_holdings
```

---

## 5) 장애/이상 대응(즉시 중단 절차)

### 5.1 계좌 단위 자동매매 끄기(가장 안전)

- UI 또는 API로 해당 계좌를 disable:
  - `POST /v1/investments/accounts/{accountId}/auto-trade/disable`

### 5.2 Celery 중단

```bash
cd /root/self-host-finance
kill $(cat logs/celery-worker.pid)
kill $(cat logs/celery-beat.pid)
```

### 5.3 로그 확인 위치

- `logs/celery-worker.log`
- `logs/celery-beat.log`
- `logs/backend.log`

---

## 6) “안 팔리는” 경우 빠른 체크

- `broker_accounts.order_enabled = true` 인가?
- `broker_accounts.auto_trade_enabled = true` 인가?
- 규칙의 `side=sell` 인가?
- `investment_holding_snapshot`에 해당 종목의 `current_price/average_price/quantity`가 0이 아닌가?
- `auto_trade_run_logs`에 `failed`가 쌓이는가? (실패 사유 확인)
- KIS 토큰/키가 실전 값인가? (`KIS_BASE_URL` 포함)
