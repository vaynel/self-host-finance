# Python 백엔드 프레임워크 비교: Flask vs FastAPI vs Django

> FinFlow 백엔드 선택을 위한 비교 문서  
> 최종 수정일: 2026-03-12

---

## 1. 한 줄 요약

| 프레임워크 | 한 줄 요약 |
|------------|------------|
| **Flask** | 가볍고 유연한 마이크로 프레임워크. 필요한 걸 직접 골라 쓴다. |
| **FastAPI** | **API 전용**, 타입·검증·문서 자동화에 강한 현대적 비동기 프레임워크. |
| **Django** | 관리자·인증·ORM까지 갖춘 올인원 풀스택 프레임워크. |

---

## 2. 상세 비교

### 2.1 Flask

**특징**
- **마이크로**: 핵심만 제공하고, 라우팅·요청/응답 처리 위주. DB·인증·검증은 확장(extension)으로 추가.
- **동기(sync) 중심**: 기본이 동기. 비동기는 지원하지만 설계의 중심은 아님.
- **유연함**: 구조·라이브러리 선택을 개발자가 전부 결정.
- **검증·문서**: 기본 기능 없음. Marshmallow, flask-restx 등으로 따로 구성.

**장점**
- 학습 곡선이 완만하고, 문서·예제가 많음.
- 작은 API부터 시작하기 좋고, 점진적으로 확장 가능.
- 레거시/기존 Flask 프로젝트와의 일관성.

**단점**
- 요청 검증·API 문서·의존성 주입 등을 직접 설계하거나 확장으로 붙여야 함.
- 비동기·고성능이 목표라면 FastAPI가 더 적합.

**적합한 경우**
- 소규모 API, 프로토타입, “원하는 대로 다 직접 맞추고 싶을 때”.

---

### 2.2 FastAPI

**특징**
- **API 우선**: REST/JSON API 구축에 맞춰 설계됨.
- **Pydantic 내장**: 요청 body/query를 스키마로 정의하면 자동 검증 + 타입 힌트.
- **자동 OpenAPI(Swagger)**: 라우트만 정의해도 `/docs`, `/redoc`에서 API 문서 생성.
- **비동기(async/await)**: I/O 많은 작업(DB, 파일, 외부 API)에 유리.
- **의존성 주입**: `Depends()`로 인증·DB 세션 등을 깔끔하게 주입.

**장점**
- API 스펙(요청/응답 형식)이 코드와 문서로 한 번에 관리됨 → FinFlow처럼 **API 정의서가 있는 프로젝트**와 잘 맞음.
- 타입 힌트 + Pydantic으로 런타임 검증과 IDE 지원이 동시에 가능.
- 성능이 좋고(Uvicorn ASGI), 비동기 DB(SQLAlchemy 2 async)와 자연스럽게 연동.

**단점**
- Django처럼 관리자 화면·풀스택 기능은 없음. “API 서버만” 필요할 때는 오히려 장점.
- 상대적으로 역사가 짧아, 레거시 예제는 Flask/Django보다 적음.

**적합한 경우**
- **순수 REST API 서버**(프론트는 React 등 별도) ← **FinFlow와 동일한 구조**.
- 요청/응답 형식을 엄격히 맞추고, 문서를 자동으로 유지하고 싶을 때.

---

### 2.3 Django (+ Django REST Framework, DRF)

**특징**
- **풀스택**: ORM, 마이그레이션, 관리자(admin), 인증(세션·그룹·권한), 폼, 템플릿 등이 기본 포함.
- **DRF**: REST API는 Django REST Framework로 구현. 시리얼라이저로 요청/응답 정의, ViewSet으로 CRUD 간소화.
- **동기 중심**: Django 4.1+에서 async 뷰 지원하지만, ORM·생태계는 여전히 동기 사용이 주류.
- **규모감**: 프로젝트 구조·설정이 정해져 있어 “Django 방식”을 따르게 됨.

**장점**
- 관리자 화면이 필요하면 즉시 사용 가능(거래/계좌 등을 admin에서 조회·수정).
- 인증·권한·모델 설계가 프레임워크에 녹아 있어, 규모가 큰 팀·장기 프로젝트에서 일관성 유지에 유리.
- 문서·책·커뮤니티가 많음.

**단점**
- “API만” 필요할 때는 기능이 과할 수 있음. FinFlow는 React가 UI를 담당하므로 Django의 템플릿·폼은 사용하지 않음.
- JWT는 Django 기본이 아니라서, `djangorestframework-simplejwt` 등 별도 구성 필요.
- 구조·학습량이 Flask/FastAPI보다 무겁다.

**적합한 경우**
- 관리자 페이지를 Django admin으로 빠르게 쓰고 싶을 때.
- 팀이 Django에 익숙하거나, 장기적으로 대규모 백오피스/풀스택을 염두에 둘 때.

## 3. FinFlow 프로젝트에 어울리는 선택: **FastAPI**

### 3.1 FinFlow 백엔드가 해야 할 일

- **REST API만** 제공 (화면은 React SPA가 담당).
- 인증: JWT(access/refresh), 회원가입/로그인.
- 리소스: 거래·계좌·투자·리포트·설정 CRUD 및 집계.
- 파일 업로드: CSV/Excel 파싱 후 거래 일괄 등록.
- **API 스펙이 이미 정의됨** (`frontend/docs/API정의서.md`).

즉, “풀스택 웹앱”이 아니라 **“JSON API 서버”**가 필요합니다.

### 3.2 FastAPI가 더 어울리는 이유

| 기준 | FastAPI | Flask | Django |
|------|---------|--------|--------|
| **API 전용 설계** | ✅ API·OpenAPI가 핵심 | 보조(확장 필요) | DRF로 가능하나 풀스택과 혼재 |
| **요청/응답 검증** | ✅ Pydantic 내장, API 정의서와 1:1 매핑 용이 | 별도 라이브러리 | DRF Serializer |
| **API 문서 자동 생성** | ✅ `/docs` 기본 제공 | 별도 구성 | DRF로 가능 |
| **비동기(DB·파일·외부 API)** | ✅ async 네이티브 | 제한적 | 지원하나 동기 사용이 주류 |
| **구성의 가벼움** | ✅ API에 필요한 것만 조합 | ✅ 가벼움 | ❌ admin·ORM 등 전체 포함 |
| **JWT·커스텀 인증** | ✅ `Depends()`로 깔끔하게 구현 | 확장으로 구현 | 패키지로 가능, 구조는 Django 스타일 |

- **API 정의서와의 정합성**: Pydantic 스키마를 정의서 필드와 맞춰 두면, 검증과 문서가 동시에 맞춰져서 프론트와 계약이 틀어질 가능성이 줄어듦.
- **관리자 화면이 필수가 아님**: 지금 기획은 “React 대시보드 + API”이므로, Django admin이 꼭 필요하지 않음. 나중에 간단한 백오피스가 필요하면 FastAPI 위에 별도 admin 라우트를 두거나, 다른 도구를 붙여도 됨.
- **성능·확장**: 비동기로 DB·파일 처리 시 동시 요청 처리에 유리하고, 나중에 트래픽이 늘어나도 대응하기 좋음.

### 3.3 다른 선택이 나을 수 있는 경우

- **Flask**: 이미 Flask에 익숙하고, “최소한의 구조로 빠르게 프로토타입만 만들고 싶다”면 선택 가능. 다만 검증·문서·비동기는 직접 설계해야 함.
- **Django + DRF**: “내부용 관리자 화면을 Django admin으로 바로 쓰고 싶다”, “팀이 Django를 이미 쓴다”면 선택 가능. 대신 JWT·API 중심 설계는 직접 맞춰야 함.

---

## 4. 정리

- **Flask**: 가볍고 유연. 작은 API나 프로토타입, “내가 다 설계하고 싶을 때”.
- **FastAPI**: **API 전용**, 검증·문서·비동기 내장. **FinFlow처럼 “정의된 스펙의 REST API만” 필요할 때 가장 잘 맞음.**
- **Django**: 관리자·풀스택·대규모 팀 일관성이 중요할 때.

**FinFlow에는 FastAPI를 추천**하며, 현재 `docs/백엔드_개발문서.md`도 FastAPI 기준으로 작성되어 있습니다.
