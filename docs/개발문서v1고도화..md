# 자산운용 웹 개발 문서

## 1. 문서 개요

### 1.1 목적

본 문서는 FastAPI 기반 자산운용 웹에서 다음 기능을 설계하기 위한 개발 문서이다.

* 일반 계좌(현금 계좌) 관리
* 주식 계좌 관리
* 증권사 연동 기반 시세/잔고 조회
* 자동매매 옵션 설계
* 엑셀 업로드 기반 현금 유동 데이터 반영
* 향후 다중 증권사 확장 가능한 구조 설계

### 1.2 현재 전제

* 백엔드: FastAPI
* 현재 주식 API 연동 대상: 한국투자증권 Open API
* 자동매매는 증권사별 지원 범위가 다르므로 옵션화 필요
* 일반 계좌 현금 유동은 현재 엑셀 업로드 방식으로 관리
* 목표는 자산 전체(현금 + 주식 + 향후 기타 자산)를 통합 관리하는 웹 서비스 구축

---

## 2. 개발 목표

### 2.1 핵심 목표

1. 사용자의 일반 계좌와 주식 계좌를 통합 관리한다.
2. 계좌별 현금 흐름과 자산 평가금액을 한 화면에서 조회할 수 있어야 한다.
3. 주식 계좌는 증권사 API 연동 여부에 따라 수동형/자동형으로 구분하여 운영한다.
4. 자동매매 기능은 계좌별 옵션으로 활성/비활성 설정할 수 있어야 한다.
5. 일반 계좌 현금 유동은 엑셀 업로드로 시작하되, 향후 자동 수집 구조로 확장 가능해야 한다.

### 2.2 비목표

* 초단타 실시간 체결 엔진 수준의 거래 시스템 구현
* 모든 증권사 API 즉시 지원
* 회계 시스템 수준의 복식부기 전면 구현

## 3. 전체 서비스 범위

### 3.1 관리 대상 자산

* 일반 계좌

  * 입출금 계좌
  * 생활비 계좌
  * 예비비 계좌
* 주식 계좌

  * 국내주식 계좌
  * 해외주식 계좌
* 확장 예정 자산

  * 예금/적금
  * 연금
  * 가상자산
  * 카드 사용 내역

### 3.2 계좌 유형 정의

#### 일반 계좌

은행 계좌 또는 현금성 계좌를 의미하며, 수입/지출/이체/기타 현금 흐름을 중심으로 관리한다.

#### 주식 계좌

증권 계좌를 의미하며, 보유 종목, 주문 내역, 체결 내역, 평가금액, 수익률을 중심으로 관리한다.

## 6. 주식계좌 설계

## 6.1 핵심 개념

주식계좌는 단순 보유 조회용 계좌와 자동매매 가능 계좌로 나뉜다.

### 유형

* VIEW_ONLY: 조회 전용
* MANUAL_TRADE: 수동 주문 보조
* AUTO_TRADE: 자동매매 허용

## 6.2 증권사 지원 정책

초기 버전에서는 한국투자증권만 API 기반 연동 대상으로 한다.
다른 증권사는 다음과 같이 구분한다.

* API 연동 가능: 향후 추가
* 수동 입력/엑셀 기반: 임시 지원
* 자동매매 미지원: 정책적으로 제한

## 6.3 증권사 옵션화 필요성

모든 증권사가 동일한 기능을 지원하지 않기 때문에, 주식계좌 기능은 계좌 단위 옵션으로 분리해야 한다.

### 계좌 옵션 예시

* broker_type: KIS / MANUAL / OTHER
* can_sync_balance: true/false
* can_sync_price: true/false
* can_place_order: true/false
* can_auto_trade: true/false
* is_active: true/false

### 정책 예시

* 한국투자증권 계좌: 조회 가능, 주문 가능, 자동매매 가능
* 수동 입력 증권계좌: 조회 수동, 주문 불가, 자동매매 불가

## 7. 자동매매 옵션 설계

## 7.1 목표

자동매매는 모든 계좌에 강제로 붙는 기능이 아니라, 사용자가 명시적으로 설정한 계좌/종목에만 동작해야 한다.

## 7.2 자동매매 설정 단위

1. 계좌 단위
2. 종목 단위
3. 규칙 단위

## 7.3 계좌 단위 옵션

* 자동매매 활성 여부
* 거래 가능 시간대
* 일일 최대 주문 횟수
* 일일 최대 주문 금액
* 주문 실패 시 중단 여부

## 7.4 종목 단위 옵션

* 감시 여부
* 목표가
* 손절가
* 분할매도 비율
* 전량매도 여부
* 1회 주문 수량
* 재주문 허용 여부

## 7.5 규칙 예시

### 목표가 매도

* 현재가 >= 목표가
* 보유 수량 > 0
* 당일 동일 규칙 미실행
* 주문 가능 시 매도

### 손절가 매도

* 현재가 <= 손절가
* 보유 수량 > 0
* 장중 여부 확인
* 주문 가능 시 매도

### 분할매도

* 1차 목표가 도달 시 30%
* 2차 목표가 도달 시 30%
* 3차 목표가 도달 시 잔량 매도

## 7.6 안전장치

* 주문 전 계좌 상태 재확인
* 매도 가능 수량 확인
* 중복 주문 방지 락
* 장 마감 후 주문 방지
* 테스트 모드 지원
* 실계좌/모의계좌 구분
* 관리자 로그 기록

## 8. 일반 계좌 현금 유동 설계

## 8.1 목표

일반 계좌는 사용자의 실질적인 현금 흐름을 관리하는 핵심 영역이다. 현재는 엑셀 업로드 기반으로 데이터를 적재하며, 향후 API 또는 자동 수집 방식으로 확장 가능해야 한다.

## 8.2 관리 항목

* 입금
* 출금
* 계좌 간 이체
* 자동이체
* 카드 대금 출금
* 현금 조정
* 수동 정산

## 8.3 입력 방식

### 현재 방식

* 사용자가 은행 거래내역 엑셀 파일 업로드
* 업로드된 파일을 파싱하여 거래내역 생성

### 향후 확장

* 은행 API
* 오픈뱅킹
* 이메일/문서 자동 파싱
* OCR 업로드

## 8.4 엑셀 업로드 처리 흐름

```text
엑셀 업로드
→ 파일 형식 검증
→ 컬럼 매핑
→ 거래내역 파싱
→ 중복 체크
→ 계좌 매핑
→ 거래 유형 분류
→ DB 저장
→ 업로드 결과 리포트 반환
```

## 8.5 현금 거래 유형 분류

* INCOME: 수입
* EXPENSE: 지출
* TRANSFER_IN: 이체입금
* TRANSFER_OUT: 이체출금
* ADJUSTMENT: 수동조정
* FEE: 수수료
* INTEREST: 이자
* DIVIDEND_CASH: 배당금

## 8.6 일반 계좌 설계 원칙

* 원본 업로드 파일은 보존한다.
* 파싱 결과와 원본을 연결한다.
* 중복 업로드를 식별할 수 있어야 한다.
* 계좌별 현금 잔액 재계산 가능해야 한다.
* 거래 분류는 수동 수정 가능해야 한다.

## 12. 한국투자증권 연동 설계

## 12.1 초기 지원 범위

* 계좌 잔고 조회
* 보유 종목 조회
* 현재가 조회
* 주문 가능 수량 조회
* 매도 주문
* 주문/체결 조회

## 12.2 연동 방식

* OAuth 토큰 발급
* API 호출용 어댑터 계층 분리
* 계좌별 API 사용 가능 여부 저장
* 실계좌/모의계좌 구분 컬럼 보유

## 12.3 어댑터 인터페이스 예시

```python
class BrokerAdapter:
    async def get_balance(self, account):
        pass

    async def get_holdings(self, account):
        pass

    async def get_current_price(self, symbol, market):
        pass

    async def place_sell_order(self, account, symbol, qty, price=None):
        pass
```

이 구조로 만들면 향후 다른 증권사 API가 추가될 때도 동일 인터페이스로 확장 가능하다.

## 13. 엑셀 기반 일반계좌 설계 상세

* 거래일시
* 입출금 구분
* 거래금액
* 거래 후 잔액
* 거래처명
* 적요
* 메모

## 13.2 컬럼 매핑 전략

은행별 엑셀 형식이 다를 수 있으므로, 업로드 시 컬럼 매핑 설정을 둘 수 있어야 한다.

예시:

* 거래일자 → transaction_date
* 출금금액 → amount + EXPENSE
* 입금금액 → amount + INCOME
* 내용 → description
* 잔액 → balance_after

## 13.3 중복 방지 전략

다음 조합으로 중복 후보를 판별한다.

* account_id
* transaction_date
* amount
* description
* balance_after

필요 시 해시값을 별도로 저장한다.

## 17. 결론

본 시스템은 단순 가계부가 아니라, 일반 계좌의 현금 유동과 주식 계좌의 투자 운용을 통합하는 자산운용 웹을 목표로 한다.

초기에는 다음 원칙으로 개발하는 것이 적절하다.

1. 일반 계좌는 엑셀 업로드 기반으로 안정적으로 시작한다.
2. 주식 계좌는 한국투자증권 API 연동 중심으로 시작한다.
3. 자동매매는 옵션 기능으로 분리하고, 조회 기능과 주문 기능을 명확히 구분한다.
4. 향후 증권사 확장을 고려하여 브로커 어댑터 구조로 설계한다.
5. 자산 전체 요약, 현금 흐름, 투자 손익을 하나의 대시보드로 통합한다.

이 문서를 기준으로 다음 단계에서는 DB 스키마 구체화, FastAPI 엔드포인트 상세화, 스케줄러 구현 명세, 엑셀 파서 규격 문서로 분리할 수 있다.
