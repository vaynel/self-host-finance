# 투자 고도화 운영 배포 체크리스트

이 문서는 `v1.1 ~ v1.10` 고도화 이후, 실제 운영 가능한 수준으로 안정화하기 위한 실행 가이드입니다.

---

## 1) 배포 전 필수 점검

### 1.1 환경변수 점검

- `KIS_APP_KEY`, `KIS_APP_SECRET` 운영값 설정
- `TOKEN_ENCRYPTION_KEY` 설정 (미설정 시 `JWT_SECRET` fallback 사용 중단 권장)
- `DATABASE_URL`, `JWT_SECRET`, `CORS_ORIGIN` 운영 환경값 확인
- 로그에 토큰/계좌번호 원문이 남지 않도록 마스킹 정책 확인

### 1.2 DB 마이그레이션 점검

```bash
cd /root/self-host-finance/backend
alembic current
alembic upgrade head
alembic current
```

확인 대상 테이블:

- `broker_accounts`
- `broker_tokens`
- `investment_holding_snapshot`
- `investment_cash_snapshot`
- `investment_orders`
- `investment_executions`
- `auto_trade_rules`
- `auto_trade_run_logs`

### 1.3 서버 기동/헬스 확인

```bash
cd /root/self-host-finance/backend
uvicorn app.main:app --host 0.0.0.0 --port 8001
```

- `GET /health` 확인
- startup 시 백그라운드 루프 기동 로그 확인
  - price updater
  - order status scheduler

## 2) E2E 검증 순서 (모의투자 -> 실전투자 순)

### 2.1 계좌/연동 준비

1. 투자 계좌 생성 (`POST /v1/accounts`)
2. KIS 연동(connect) 수행해서 `broker_accounts` 생성
   - UI: `투자 관리` → 계좌별 `KIS 연동`(CANO/상품코드 입력)
   - API: `POST /v1/investments/accounts/{accountId}/kis/connect`
3. KIS 토큰 발급/갱신 정상 동작 확인

### 2.2 holdings / cash 동기화 검증

1. `POST /v1/investments/accounts/{accountId}/sync`
2. `GET /v1/investments/holdings`
3. `source` 필드가 `snapshot`으로 내려오는지 확인
4. KIS 실패 시 fallback(`trade_fallback`) 동작 여부 확인

### 2.3 주문/체결/정산 검증

1. `POST /v1/investments/orders`
2. `POST /v1/investments/orders/{orderId}/refresh`
3. `GET /v1/investments/orders/{orderId}`
4. 체결 발생 시 아래가 동시에 반영되는지 확인
   - `investment_executions` 저장
   - `investment_trades` 생성
   - `transactions` 생성
   - `accounts.balance` 증감

### 2.4 idempotency 검증

동일 주문에 대해 refresh를 연속 호출해도:

- execution 중복 저장이 없어야 함
- 정산 중복 반영이 없어야 함

## 3) 자동매매 운영 점검

### 3.1 규칙 CRUD

- `GET /v1/investments/rules`
- `POST /v1/investments/rules`
- `PUT /v1/investments/rules/{ruleId}`
- `DELETE /v1/investments/rules/{ruleId}`

### 3.2 계좌별 자동매매 on/off

- `POST /v1/investments/accounts/{accountId}/auto-trade/enable`
- `POST /v1/investments/accounts/{accountId}/auto-trade/disable`

### 3.3 가드 로직 점검

- 장중 체크 (평일 09:00~15:30 KST)
- 쿨다운 (`cooldown_seconds`)
- 중복 pending 주문 방지

---

## 4) 장애/이상 상황 대응 절차

### 4.1 KIS API 장애

- 증상: 토큰 발급 실패, quote/order/status 조회 실패
- 조치:
  1. KIS API 상태/키 유효성 확인
  2. 가격은 기존 캐시 유지 (쓰기 실패 시 skip)
  3. 주문 상태 폴링 실패 로그 모니터링
  4. 필요 시 자동매매 전 계좌 disable

### 4.2 정산 불일치

- 증상: execution 존재하지만 balance/ledger 반영 불일치
- 조치:
  1. `investment_executions.settled` 상태 확인
  2. 동일 `broker_execution_id` 중복 여부 확인
  3. 실패건 재처리 스크립트 또는 수동 정산 실행

### 4.3 스케줄러 미동작

- 증상: 가격/주문 상태 갱신이 멈춤
- 조치:
  1. 앱 startup 로그 확인
  2. 프로세스 재기동
  3. loop exception 로그 수집 후 원인 제거

---

## 5) 운영 모니터링 권장 항목

- 주문 상태별 카운트 (`pending`, `partially_filled`, `failed`)
- 정산 실패 카운트 (`settled=error`)
- 가격 갱신 실패율 (ticker/user별)
- 자동매매 rule trigger/skip/fail 비율

로그 공통 키 권장:

- `user_id`
- `account_id`
- `order_id`
- `execution_id`
- `rule_id`

---

## 6) 프론트 연동 최종 점검

- holdings `source` 배지 표시 (`snapshot` / `trade_fallback`)
- 주문 상세 화면에서 execution/settled 상태 표시
- 자동매매 규칙 CRUD 및 계좌 on/off 토글 연결
- 에러 메시지 표준화 (장외시간, 잔고부족, KIS 장애)

---

## 7) 배포 후 체크 (Day 1)

배포 후 24시간 동안 아래를 집중 확인:

1. 신규 주문의 체결/정산 반영 지연
2. 자동매매 규칙 오동작 여부
3. holdings snapshot 갱신 누락 여부
4. 수동/자동 주문 경로 간 원장 불일치 여부

문제 발생 시:

- 자동매매 우선 disable
- 주문 상태 폴링 유지
- 정산 무결성부터 복구 후 기능 재개
