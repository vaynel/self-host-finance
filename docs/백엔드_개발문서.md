## 1. 개요

### 1.1 역할

FinFlow 백엔드는 개인 재무 관리 플랫폼의 **REST API 서버**로, 다음을 담당합니다.

- 사용자 인증(로그인/회원가입/토큰 관리)
- 거래·계좌·투자·리포트 데이터 CRUD 및 집계
- CSV/Excel 업로드 기반 거래 일괄 등록
- 사용자별 데이터 격리 및 보안

### 1.2 프론트엔드와의 관계

- **클라이언트**: React SPA (Vite + TanStack Query)
- **통신**: REST API, JSON, Bearer JWT
- **계약**: API 스펙은 `frontend/docs/API정의서.md` 기준으로 구현

## 2. 기술 스택 권장 (Python)

| 분류 | 기술 | 비고 |
|------|------|------|
| 런타임 | Python 3.11+ | 3.12 권장 |
| 프레임워크 | **FastAPI** | REST API, 자동 OpenAPI, 비동기 지원. 대안: Django + DRF |
| DB | PostgreSQL 15+ | 메인 저장소 |
| ORM | **SQLAlchemy 2.x** (async) / Alembic | FastAPI와 연동. Django 사용 시 Django ORM |
| 인증 | **PyJWT** + **passlib[bcrypt]** | JWT 발급/검증, 비밀번호 해시 |
| 검증 | **Pydantic** (FastAPI 내장) | 요청 body/query 스키마 및 검증 |
| 파일 업로드 | FastAPI `UploadFile` + **openpyxl** / **pandas** | CSV/Excel 파싱 |
| 서버 | **Uvicorn** (ASGI) | FastAPI 실행용 |
| 테스트 | **pytest** + **httpx** (TestClient) | 단위·통합·API 테스트 |

## 4. 환경 변수

`.env`는 버전 관리에서 제외하고, `.env.example`만 커밋합니다.

| 변수명 | 필수 | 설명 | 예시 |
|--------|------|------|------|
| `DATABASE_URL` | ✅ | PostgreSQL 연결 문자열 | `postgresql://user:pass@localhost:5432/finflow` |
| `JWT_SECRET` | ✅ | JWT 서명용 비밀키 | 32자 이상 랜덤 문자열 |
| `JWT_ACCESS_EXPIRES_IN` | - | Access Token 만료 | `1h` (기본) |
| `JWT_REFRESH_EXPIRES_IN` | - | Refresh Token 만료 | `7d` (기본) |
| `PORT` | - | 서버 포트 | `8000` |
| `ENV` | - | `development` / `staging` / `production` | `development` |
| `CORS_ORIGIN` | - | 허용 프론트엔드 오리진 | `http://localhost:5173` |

현재 `backend/.env` 예시:

```env
DATABASE_URL=postgresql://mandu:mandu@localhost:5432/mandu
```

운영 환경에서는 `JWT_SECRET`, `CORS_ORIGIN` 등을 반드시 설정합니다.

## 5. 데이터베이스 스키마

개발기획문서의 데이터 모델을 기준으로 합니다. ID는 UUID 사용을 권장합니다.

### 5.1 users

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| id | UUID | PK, default uuid_generate_v4() | 고유 식별자 |
| email | VARCHAR(255) | UNIQUE, NOT NULL | 이메일 |
| name | VARCHAR(100) | NOT NULL | 사용자 이름 |
| password_hash | VARCHAR(255) | NOT NULL | bcrypt 해시 |
| created_at | TIMESTAMPTZ | NOT NULL, default now() | 가입일 |
| updated_at | TIMESTAMPTZ | NOT NULL, default now() | 수정일 |

### 5.2 refresh_tokens (선택)

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| id | UUID | PK | 토큰 식별자 |
| user_id | UUID | FK(users), NOT NULL | 사용자 |
| token | VARCHAR(512) | UNIQUE, NOT NULL | 리프레시 토큰 값 |
| expires_at | TIMESTAMPTZ | NOT NULL | 만료 시각 |
| created_at | TIMESTAMPTZ | NOT NULL | 발급 시각 |

### 5.3 transactions

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| id | UUID | PK | 고유 식별자 |
| user_id | UUID | FK(users), NOT NULL | 사용자 |
| date | DATE | NOT NULL | 거래 날짜 |
| description | VARCHAR(500) | NOT NULL | 거래 내용 |
| amount | DECIMAL(15,2) | NOT NULL | 금액 (지출은 음수) |
| type | VARCHAR(20) | NOT NULL | `income` / `expense` / `transfer` |
| category | VARCHAR(100) | NOT NULL | 카테고리 |
| account | VARCHAR(100) | NOT NULL | 계좌명 |
| memo | TEXT | - | 메모 |
| created_at | TIMESTAMPTZ | NOT NULL | 생성일 |
| updated_at | TIMESTAMPTZ | NOT NULL | 수정일 |

- `user_id` + `date` 등에 인덱스 권장 (목록/필터 조회용).

### 5.4 accounts

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| id | UUID | PK | 고유 식별자 |
| user_id | UUID | FK(users), NOT NULL | 사용자 |
| name | VARCHAR(100) | NOT NULL | 계좌 별칭 |
| type | VARCHAR(20) | NOT NULL | `bank` / `investment` |
| balance | DECIMAL(15,2) | NOT NULL | 현재 잔액 |
| institution | VARCHAR(100) | NOT NULL | 금융기관명 |
| last_sync | TIMESTAMPTZ | - | 마지막 동기화 시각 |
| created_at | TIMESTAMPTZ | NOT NULL | 생성일 |
| updated_at | TIMESTAMPTZ | NOT NULL | 수정일 |

### 5.5 investment_trades

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| id | UUID | PK | 고유 식별자 |
| user_id | UUID | FK(users), NOT NULL | 사용자 |
| ticker | VARCHAR(20) | NOT NULL | 종목 코드 |
| name | VARCHAR(100) | - | 종목명 |
| type | VARCHAR(20) | NOT NULL | `stock` / `etf` |
| action | VARCHAR(10) | NOT NULL | `buy` / `sell` |
| date | DATE | NOT NULL | 거래 날짜 |
| shares | DECIMAL(18,4) | NOT NULL | 수량 |
| price | DECIMAL(15,2) | NOT NULL | 단가 |
| fee | DECIMAL(15,2) | - | 수수료 |
| created_at | TIMESTAMPTZ | NOT NULL | 생성일 |

### 5.6 user_settings

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| user_id | UUID | PK, FK(users) | 사용자 |
| currency | VARCHAR(3) | default 'KRW' | 기본 통화 |
| language | VARCHAR(5) | default 'ko' | 언어 |
| notifications | JSONB | - | 알림 설정 |
| updated_at | TIMESTAMPTZ | NOT NULL | 수정일 |

- Row Level Security(RLS) 또는 애플리케이션 레벨에서 **반드시 user_id로 필터링**하여 사용자별 데이터 격리를 보장합니다.

## 6. 공통 응답·에러 형식

API 정의서와 동일하게 맞춥니다.

### 6.1 성공 응답

```json
{
  "success": true,
  "data": { ... },
  "error": null,
  "meta": {
    "page": 1,
    "limit": 20,
    "total": 100
  }
}
```

- 목록 API: `data`는 배열, `meta`에 페이지 정보.
- 단건 API: `data`는 객체, `meta` 생략 가능.

### 6.2 에러 응답

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": 400,
    "message": "유효성 검증 실패",
    "details": [ "amount는 숫자여야 합니다." ]
  },
  "meta": null
}
```

### 6.3 HTTP 상태 코드

| 코드 | 용도 |
|------|------|
| 200 | 성공 (GET/PUT 등) |
| 201 | 생성 성공 (POST) |
| 400 | 잘못된 요청(유효성 검증 실패) |
| 401 | 인증 실패(토큰 만료/미제공) |
| 403 | 권한 없음 |
| 404 | 리소스를 찾을 수 없음 |
| 500 | 서버 내부 오류 |

## 7. 인증

### 7.1 흐름

- **로그인**: 이메일·비밀번호 검증 → Access Token + Refresh Token 발급.
- **인증 필요 API**: `Authorization: Bearer <accessToken>` 필수.
- **토큰 갱신**: `POST /auth/refresh`에 `refreshToken` 전달 → 새 Access Token( 및 선택적으로 Refresh Token) 발급.
- **로그아웃**: Refresh Token 무효화(블랙리스트 또는 DB에서 삭제).

### 7.2 보안 요구사항 (기획문서 기준)

- 비밀번호: **bcrypt** 해시 후 저장 (salt round 10~12 권장).
- Access Token 만료: **1시간**.
- Refresh Token 만료: **7일**.
- HTTPS(TLS 1.3) 사용.
- SQL Injection / XSS / CSRF 방지 (CORS, Helmet, 입력 검증 등).

### 7.3 인증 의존성 (FastAPI)

- `Authorization` 헤더에서 Bearer 토큰 추출 → 검증 후 `get_current_user` 의존성에서 `User` 객체 반환.
- 토큰 없음/만료/위조 → 401 JSON 응답 (HTTPException).
- 리소스 소유자 확인: 예) `transactions` 수정/삭제 시 `transaction.user_id == current_user.id` 검사.

## 8. API 구현 체크리스트

아래는 `frontend/docs/API정의서.md`와 대응하는 엔드포인트별 구현 시 확인할 사항입니다.

### 8.1 인증

- [ ] `POST /auth/login` — 이메일·비밀번호 검증, JWT 발급, 응답 형식 준수.
- [ ] `POST /auth/register` — 이메일 중복 체크, 비밀번호 정책(최소 8자, 영문+숫자+특수문자), 해시 저장.
- [ ] `POST /auth/refresh` — refreshToken 검증 후 새 accessToken 발급.
- [ ] `POST /auth/logout` — Bearer 필수, refreshToken 무효화.

### 8.2 거래 (transactions)

- [ ] `GET /transactions` — 쿼리: page, limit, type, category, account, startDate, endDate, search. user_id 필터 필수.
- [ ] `GET /transactions/:id` — 본인 거래만 200, 아니면 403/404.
- [ ] `POST /transactions` — body 검증, user_id 자동 설정.
- [ ] `PUT /transactions/:id` — 본인 것만 수정.
- [ ] `DELETE /transactions/:id` — 본인 것만 삭제.

### 8.3 계좌 (accounts)

- [ ] `GET /accounts` — 해당 user의 계좌 목록.
- [ ] `POST /accounts` — name, type, balance, institution 검증.
- [ ] `PUT /accounts/:id`, `DELETE /accounts/:id` — 소유자 확인.
- [ ] `GET /accounts/:id/flow` — period(7d/30d/90d/1y)에 따른 잔액 흐름 집계 (해당 계좌 거래 기반).

### 8.4 투자 (investments)

- [ ] `GET /investments/holdings` — user별 보유 종목 집계(매수-매도), 현재가·수익률 계산(외부 API 또는 고정값).
- [ ] `GET /investments/trades` — ticker, action, startDate, endDate 필터.
- [ ] `GET /investments/prices/:ticker` — period별 시세 (외부 API 또는 mock).
- [ ] `POST /investments/trades` — ticker, action, date, shares, price, fee 검증.

### 8.5 리포트 (reports)

- [ ] `GET /reports/monthly-summary` — year 파라미터, 월별 수입/지출/저축/저축률.
- [ ] `GET /reports/category-spending` — month(YYYY-MM) 기준 카테고리별 지출.

### 8.6 업로드

- [ ] `POST /upload/transactions` — multipart/form-data, file + accountId, format(csv|xlsx). 파싱 후 유효 행만 transactions에 삽입, imported/skipped/errors 응답.

### 8.7 설정

- [ ] `GET /settings` — user_settings 조회.
- [ ] `PUT /settings` — currency, language, notifications 업데이트.

## 9. 파일 업로드 (CSV/Excel)

- **라이브러리**: FastAPI `UploadFile` + `openpyxl`(Excel), `pandas` 또는 표준 `csv` 모듈(CSV).
- **제한**: 파일 크기 제한(예: 5MB), 확장자 화이트리스트(.csv, .xlsx).
- **매핑**: 컬럼명 매핑 설정을 프론트와 협의하거나, 요청에 `mapping` JSON으로 받아서 처리.
- **트랜잭션**: 일괄 삽입은 DB 세션 트랜잭션으로 묶어 부분 실패 시 롤백 또는 오류 행만 건너뛰고 결과 집계.

## 10. 테스트

- **단위**: 서비스·유틸 함수 (비밀번호 해시, JWT 생성/검증, 집계 로직 등) — `pytest`.
- **통합**: API 엔드포인트 — FastAPI `TestClient`(httpx) + 테스트 DB 또는 SQLite in-memory.
- **필수 시나리오**: 로그인 실패/성공, 인증 없이 보호된 API 호출(401), 타 사용자 리소스 접근(403), 거래 CRUD, 계좌 flow, 리포트 집계.

## 11. 배포·운영

- **실행**: `uvicorn app.main:app --host 0.0.0.0 --port 8000` (개발 시 `--reload`).
- **프로세스 매니저**: Gunicorn + Uvicorn worker, 또는 systemd/supervisor.
- **환경**: `ENV=production`, `DATABASE_URL`, `JWT_SECRET`, `CORS_ORIGIN` 등 필수 설정.
- **마이그레이션**: 배포 전 `alembic upgrade head` 로 스키마 적용.
- **헬스 체크**: `GET /health` (DB 연결 확인 포함 권장).
